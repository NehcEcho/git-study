## 6.1 撤销操作概览

Git 提供了多种撤销操作，适用于不同场景：

| 场景 | 命令 |
|------|------|
| 撤销工作区的修改 | `git checkout -- <file>` |
| 撤销暂存区的文件 | `git reset HEAD <file>` |
| 修改最后一次提交 | `git commit --amend` |
| 回退到指定版本 | `git reset --hard <commit>` |
| 查看所有操作 | `git reflog` |

---

## 6.2 撤销工作区的修改

### 场景：修改了文件，但还没 add

```bash
# 创建并提交一个文件
echo "原始内容" > demo.txt
git add demo.txt
git commit -m "添加 demo.txt"

# 修改文件
echo "错误的内容" > demo.txt

# 查看状态
git status
```

### ✅ 预期输出

```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   demo.txt

no changes added to commit (use "git add" to add)
```

### 撤销修改

```bash
# 丢弃工作区的修改（危险：修改会丢失！）
git checkout -- demo.txt

# 或者使用新命令（Git 2.23+）
git restore demo.txt
```

### 验证撤销结果

```bash
cat demo.txt
```

### ✅ 预期输出

```
原始内容
```

> 💡 文件恢复到了最后一次 commit 的状态

---

## 6.3 撤销暂存区的文件

### 场景：add 了文件，但还没 commit

```bash
# 修改文件
echo "新内容" > demo.txt

# 添加到暂存区
git add demo.txt

# 查看状态
git status
```

### ✅ 预期输出

```
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        modified:   demo.txt
```

### 撤销暂存

```bash
# 将文件从暂存区移回工作区
git reset HEAD demo.txt

# 或者使用新命令
git restore --staged demo.txt
```

### ✅ 预期输出

```
Unstaged changes after reset:
M	demo.txt
```

### 查看状态

```bash
git status
```

### ✅ 预期输出

```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   demo.txt
```

> 💡 文件现在在工作区，修改仍然保留

## 6.4 修改最后一次提交

### 场景：提交了，但发现有问题

```bash
# 提交文件
git add demo.txt
git commit -m "更新 demo.txt"

# 哎呀，提交信息写错了，或者漏了文件
```

### 修改提交信息

```bash
# 修改最后一次提交的信息
git commit --amend -m "正确的提交信息"
```

### 添加漏掉的文件

```bash
# 创建另一个文件
echo "补充内容" > extra.txt
git add extra.txt

# 合并到上一次提交
git commit --amend -m "更新 demo.txt 并添加 extra.txt"
```

> ⚠️ 注意：如果已经 push 到远程，不要 amend，否则会造成混乱！

## 6.5 使用 reflog 恢复

`reflog` 记录了所有的操作历史，是后悔药！

```bash
# 查看所有操作记录
git reflog
```

### ✅ 预期输出

```
abc1234 (HEAD -> main) HEAD@{0}: commit (amend): 更新 demo.txt 并添加 extra.txt
def5678 HEAD@{1}: commit: 更新 demo.txt
ghi7890 HEAD@{2}: commit: 添加 demo.txt
...更多记录...
```

### 恢复到任意状态

```bash
# 恢复到指定记录
git reset --hard HEAD@{2}

# 或者使用 commit id
git reset --hard ghi7890
```

## 6.6 什么是标签（Tag）？

标签是用于标记特定提交的重要工具，常用于：

- 🏷️ 标记版本号（v1.0.0, v2.0.0）
- 📌 标记发布点
- 🔖 标记里程碑

```
main:  A --- B --- C --- D --- E
                         ↑
                      v1.0.0 (tag)
```

---

## 6.7 创建标签

### 查看现有标签

```bash
# 查看所有标签
git tag

# 查看标签详情
git show v1.0.0
```

### 创建轻量标签

```bash
# 在当前提交创建轻量标签
git tag v1.0.0
```

### 创建附注标签（推荐）

```bash
# 创建带注释的标签
git tag -a v1.0.0 -m "第一个正式版本"
```

### 在指定提交创建标签

```bash
# 在历史提交上创建标签
git tag -a v0.9.0 abc1234 -m "测试版本"
```

---

## 6.8 查看和推送标签

### 查看标签

```bash
# 列出所有标签
git tag

# 列出特定模式的标签
git tag -l "v1.*"

# 查看标签详情
git show v1.0.0
```

### 推送标签到远程

```bash
# 推送单个标签
git push origin v1.0.0

# 推送所有标签
git push origin --tags
```

---

## 6.9 切换到标签

```bash
# 切换到标签指向的提交（分离 HEAD 状态）
git checkout v1.0.0

# 基于标签创建分支
git checkout -b hotfix v1.0.0
```

---

## 6.10 删除标签

```bash
# 删除本地标签
git tag -d v1.0.0

# 删除远程标签
git push origin --delete v1.0.0
```

---

## 6.11 语义化版本号

推荐使用语义化版本（Semantic Versioning）：

```
版本格式：主版本号.次版本号.修订号（MAJOR.MINOR.PATCH）

1.0.0 - 初始版本
1.1.0 - 新增功能（向后兼容）
1.1.1 - 修复 bug（向后兼容）
2.0.0 - 重大更新（可能不兼容）
```

| 版本变化 | 说明 |
|----------|------|
| MAJOR | 重大更新，可能不兼容 |
| MINOR | 新增功能，向后兼容 |
| PATCH | 修复 bug，向后兼容 |

## 6.12 常用命令速查表

| 命令 | 作用 |
|------|------|
| `git checkout -- <file>` | 撤销工作区修改 |
| `git restore <file>` | 撤销工作区修改（新命令） |
| `git reset HEAD <file>` | 撤销暂存区文件 |
| `git restore --staged <file>` | 撤销暂存区文件（新命令） |
| `git commit --amend` | 修改最后一次提交 |
| `git reflog` | 查看操作历史 |
| `git tag` | 查看标签 |
| `git tag -a <name> -m "msg"` | 创建附注标签 |
| `git push origin --tags` | 推送所有标签 |
| `git tag -d <name>` | 删除标签 |
